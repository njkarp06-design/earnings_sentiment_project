Earnings Sentiment Platform

Architecture reference — current codebase (all phases complete)

Updated June 2026

Ingests public earnings call transcripts from SEC EDGAR, scores CEO language confidence (0–100) using Claude, correlates those scores against post-call stock price returns, and surfaces everything in an interactive dashboard. The central idea: how a CEO sounds on an earnings call often predicts short-term price movement.

Seven layers from raw data sources to the user's browser. Each box is either a running service or an external dependency.

Data pipeline — one transcript, end to end

Every step a single earnings filing goes through from detection to appearing on the dashboard.

1
Detection
The SEC 8-K RSS Atom feed fires for a new filing, or the APScheduler per-ticker cron scan finds an unprocessed 8-K. Ingestor checks _ingested_filings — if this filing_id has already been processed, it stops here. An on-demand POST /trigger/<TICKER> HTTP call can also initiate the flow immediately for a single ticker.
↓
2
Fetch transcript + prices — 3-pass strategy
Pass 1 (EDGAR): Downloads the 8-K exhibit and detects transcript text via signal phrases (operator calls, Q&A language). Pass 2 (FMP): Always runs after EDGAR for structured transcript archives that go back many years; cross-source dedup via _ingested_filings and price_reactions call_date check prevents double-publishing the same call. Pass 3 (Alpha Vantage): On-demand only (25 req/day free tier) — fetches the last 8 quarters for single-ticker trigger requests. Simultaneously, yfinance fetches OHLCV data for the window around the call date. Best-effort write to S3 (AWS only).
↓
3
Publish to Kafka — two topics
Transcript text → raw-transcripts. OHLCV price dict → raw-prices. Both published simultaneously. filing_id written to _ingested_filings to prevent any future re-publish. A separate price_refresh job (every 2h) re-publishes OHLCV for any filing that still has null return fields, so the correlation service can fill them in as each trading window closes.
↓
4
Scoring service consumes raw-transcripts → calls Claude → publishes scored-transcripts
Scoring service consumes from raw-transcripts. Checks scores collection first — skips if this filing_id was already scored (idempotent). Calls Claude using a cached system prompt (ephemeral cache_control reduces API cost). Model is set by SCORING_MODEL — claude-haiku-4-5-20251001 is recommended in .env.example; docker-compose default is claude-sonnet-4-6 if no .env is present. Transcript text is hard-capped at MAX_TRANSCRIPT_CHARS (default 150k) before sending to Claude. Receives a JSON response: confidence_score (0–100) and three key_phrases. Writes the result to the scores collection, then publishes the enriched message back to Kafka on the scored-transcripts topic.
↓
5
Correlation service consumes scored-transcripts + raw-prices
The main thread consumes scored-transcripts. Fetches 12+ days of post-call OHLCV from yfinance. Calculates percentage returns at day 1, day 3, and day 7 from the baseline close (first trading day on or after the call date). Builds a price_series array (per-day {day, close, pct} used for sector drift visualisation). Looks up the company's sector from the companies collection and denormalises it onto the record. Computes a trend signal by comparing this call's score to the ticker's previous call. Writes the full denormalised record to price_reactions. If any trading window hasn't closed yet, those return fields are left null — the backfill daemon fills them in later. A separate background thread independently consumes raw-prices and caches OHLCV rows into raw_prices for the BFF chart endpoint.
↓
6
Email notifications
After writing to price_reactions, the notification system queries users for anyone who has this ticker in their watchlist and has notifications_enabled: true. Sends an HTML-formatted email via Resend API (only when NOTIFICATION_PROVIDER=resend). The _sent_notifications collection has a unique index on (user_id + ticker + call_date) — guarantees at-most-once delivery even on service restarts.
↓
7
BFF serves the call
GET /feed queries price_reactions within a 12-month window, deduplicated by (ticker, call_date) with EDGAR records preferred over FMP records, then sorted by call_date DESC. The new call appears as a feed card on the dashboard. If returns are still null (pending), a pending badge is shown and the frontend polls every 60s.
↓
8
Deep analysis on demand (streaming)
User clicks any feed card → InspectModal opens → frontend POSTs to /inspect → BFF opens a Server-Sent Events (SSE) connection to Claude Sonnet and streams a five-section deep analysis back to the browser in real time: What Happened, Key Signals, What Drove the Price, Red Flags & Green Flags, Bottom Line.

Services in detail

Python Ingestor

Three-pass transcript strategy — Pass 1: EDGAR client downloads 8-K exhibits and detects transcripts via signal phrases; respects SEC fair-use rate limits (≤10 req/s, 0.12s delay). Pass 2: FMP structured transcript archives always run after EDGAR — cross-source dedup prevents re-publishing the same call. Pass 3: Alpha Vantage (25 req/day free tier) is used only during on-demand single-ticker trigger requests to fill recent quarters.

RSS Atom poller — Polls the SEC 8-K Atom feed every RSS_POLL_INTERVAL_MINUTES (default 10 min). Catches any public company that files, even those not in the configured TICKERS list. Triggers a background historical backfill per newly discovered ticker. This is the primary real-time coverage mechanism; the per-ticker scan is a reliability backstop.

Flask HTTP trigger API (port 8001) — POST /trigger/<TICKER> initiates an immediate on-demand scan: EDGAR 90-day window + full FMP archive + Alpha Vantage 8 quarters + async full historical backfill. Called internally by the BFF /companies/:ticker/ingest endpoint. GET /health is the liveness probe.

Scheduled jobs (APScheduler) — periodic_ingest: every SCHEDULE_INTERVAL_HOURS (default 2h), runs Pass 1+2 for all tickers (static + user watchlists + recently active). price_refresh: every 2h, re-publishes OHLCV for any filing that still has null returns so the correlation service can fill them in as each trading window closes.

Watchlist expansion — On each scheduled run, reads all users' watchlists from MongoDB and merges those tickers into the scan set, so newly bookmarked companies are automatically covered.

S3 archival — Best-effort transcript backup to S3 when running in AWS. No-op locally.

Scoring Service

Claude scoring prompt — Uses a cached system prompt (ephemeral cache_control) with a five-band scoring rubric: 90–100 exceptionally confident · 70–89 confident · 50–69 neutral · 30–49 cautious · 0–29 very uncertain. Returns structured JSON. Model is configurable via SCORING_MODEL; .env.example recommends Haiku for cost efficiency; the docker-compose default when no .env is present is claude-sonnet-4-6. Transcript text is hard-capped at MAX_TRANSCRIPT_CHARS (default 150k) before being sent.

Error handling — Malformed Claude responses: logs and commits offset (skips). Network/rate-limit errors: does not commit, retries on service restart.

Output — Writes to scores collection (filing_id unique, model_used, scored_at) AND publishes enriched message to scored-transcripts Kafka topic.

Correlation Service

Three concurrent threads in one container:

  • Main thread — consumes scored-transcripts, fetches yfinance, computes 1d/3d/7d returns, builds price_series array, denormalises sector from companies collection, computes trend, writes price_reactions, fires notifications
  • Cache thread — consumes raw-prices, upserts rows into raw_prices collection for BFF chart queries
  • Backfill daemon — runs every 4 hours, finds price_reactions with any null return fields and re-fetches yfinance to fill them in (records backfilled_at); also backfills missing sector values from the companies collection

Trend signal — Compares current call's score to the same ticker's most recent previous call. Assigns "up" (≥+5), "down" (≤−5), or "neutral".

price_series — Day-by-day percentage change array stored on each price_reactions document. Used by the BFF sector endpoints to compute mean and standard-deviation drift paths across all companies in a sector.

Node.js BFF

/inspect endpoint — Accepts a POST with call metadata, opens an SSE connection, proxies Claude Sonnet's streaming response back to the browser as JSON text chunks. Rate-limited to 10 requests/hour per user. Uses INSPECT_MODEL env var (default claude-sonnet-4-6).

/companies/:ticker/ingest endpoint — Auth-required POST that forwards to the ingestor's /trigger/<TICKER> HTTP API (INGESTOR_URL, default http://ingestor:8001). Lets the frontend trigger an immediate on-demand scan for any company a user has just bookmarked.

Sector analytics — GET /sectors returns per-sector averages (confidence, returns, win rate, call count). GET /sectors/:sector returns a drift path (mean ± std per day computed across all company price_series in that sector) plus per-company rankings.

EDGAR-preferred deduplication — All aggregations (feed, portfolio, leaderboard, search) deduplicate by (ticker, call_date), preferring EDGAR records over FMP records so real company names always win over ticker-as-name FMP entries.

price_reactions as the read model — Almost all BFF queries read from this single denormalised collection. No joins across collections at request time.

MongoDB collections

price_reactions is the central read model — one denormalised document per filing containing score, returns, trend, price_series, key phrases, and sector. The BFF reads almost exclusively from this collection with no joins. Six collections are created by mongo/init/01_init.js on first startup (companies and transcripts with schema validation; the others plain). The two internal collections are created dynamically on first write.

price_reactions
Primary BFF read source. One doc per filing: score, key_phrases, 1d/3d/7d returns, trend, price_series (per-day pct array), call_date_close, sector, company_name, backfilled_at. Written by correlation-service. Unique index: filing_id.
companies
Ticker → name → sector → exchange → CIK. Seeded from EDGAR's ~10k US-listed company universe at ingestor startup. This defines what is searchable — not yfinance. International stocks, ETFs, and crypto are outside this universe.
scores
Raw scoring output per filing_id (unique). Written by scoring-service. Stores model_used and scored_at. Used only for the idempotency check before calling Claude — not queried by the BFF.
raw_prices
OHLCV cache (ticker + date unique index). Written by correlation-service's background cache thread consuming raw-prices topic. Read only by BFF /prices/:ticker chart endpoint.
users
Email, bcrypt password, watchlist array, notifications_enabled, notifications_email. Read by ingestor (ticker expansion), correlation-service (who to email), and BFF auth/portfolio routes.
transcripts
Optional raw transcript storage. The main pipeline passes text through Kafka — this collection is not in the critical path.
_ingested_filings
Internal. Created dynamically on first write (not in init script). Unique index on filing_id. Ingestor writes before publishing to Kafka — prevents re-publishing across both scheduled runs and on-demand triggers. Also used for cross-source (EDGAR/FMP) dedup by call_date.
_sent_notifications
Internal. Created dynamically on first write (not in init script). Unique compound index on (user_id, ticker, call_date). Prevents duplicate email sends on correlation-service restart or backfill re-runs.

Key design decisions

Three-pass ingest strategy, not one source

EDGAR is the authoritative source but only covers 8-K exhibit text. FMP provides structured transcript archives that go back many years and always runs as Pass 2, even when EDGAR succeeds, because they cover different historical ranges. Alpha Vantage runs only on-demand (25 req/day free tier) for single-ticker trigger requests. Cross-source dedup is handled at two levels: _ingested_filings (filing_id) and a call_date check against existing price_reactions.

Two Kafka topics from the ingestor, not one

The ingestor publishes raw-transcripts and raw-prices simultaneously. This decouples price caching (correlation-service background thread) from transcript processing (scoring pipeline). Each consumer can fail and retry independently without blocking the other. A separate price_refresh job re-publishes OHLCV every 2h for any filing with null returns, so windows fill in automatically without manual intervention.

Scoring service publishes back to Kafka rather than writing to MongoDB directly

The scoring service outputs to scored-transcripts, not straight to the database. This keeps the pipeline asynchronous: the correlation service can lag behind, replay messages, or restart without losing data. It also means the scoring and correlation stages can scale independently.

Two Claude models, two purposes

Batch scoring uses the model set by SCORING_MODEL — the recommendation in .env.example is Haiku for cost efficiency (prompt caching means the system prompt is only billed once per cache window). The docker-compose default when no .env is present is Sonnet. Sonnet via INSPECT_MODEL is used only when a user explicitly opens the InspectModal — interactive, on-demand, rate-limited to 10 req/hour.

On-demand ingest via HTTP trigger

The ingestor exposes a Flask HTTP API on port 8001. The BFF's POST /companies/:ticker/ingest forwards to it when a user bookmarks a company that has no data yet. This gives the user near-immediate feedback rather than waiting up to SCHEDULE_INTERVAL_HOURS for the next scheduled run. The trigger runs Pass 1+2+3 synchronously, then kicks off a full historical backfill asynchronously in the background.

How an unmonitored company gets data

Searching for a company does not trigger ingestion — it only reads what already exists. A company that is in the EDGAR universe but hasn't been ingested shows has_data: false in search results with no score or returns.

Adding it to a watchlist is the primary trigger. The BFF calls /companies/:ticker/ingest which hits the ingestor's HTTP trigger for immediate processing. The ingestor also reads every user's watchlist on each scheduled run and merges those tickers into its scan set. Once added:

  • Immediate HTTP trigger scan: EDGAR 90d + full FMP archive + Alpha Vantage 8 quarters
  • Async background full historical backfill (LOOKBACK_DAYS)
  • The RSS poller covers all future filings for that company in real time

The result is the same full card — score, key phrases, 1d/3d/7d returns — as any actively monitored company, typically appearing within minutes of bookmarking.

RSS poller as the real-time layer

The original design relied purely on a per-ticker APScheduler scan. The RSS 8-K Atom poller was added to give near-real-time coverage for any public company — not just the ones in the TICKERS env var. The per-ticker scan remains as a reliability backstop that catches anything the feed misses.

price_reactions as the denormalised read model

Rather than joining scores + transcripts + prices at query time, the correlation-service writes a single document per call into price_reactions containing everything the frontend needs: score, key_phrases, 1d/3d/7d returns, trend, price_series, sector, and company_name. The BFF reads almost exclusively from this one collection — no cross-collection joins on any request path.

Sector drift path via price_series

Each price_reactions document stores a price_series array: per-day percentage change from the call date baseline. The BFF /sectors/:sector endpoint aggregates all companies' arrays within a sector to compute a mean drift path and confidence band (std = √(E[X²] − E[X]²)) without storing any sector-level summary documents. This means sector charts are always live and automatically include any newly correlated calls.

Idempotency at every stage

_ingested_filings prevents re-publishing to Kafka (covers both scheduled runs and HTTP trigger calls). The scoring-service checks scores before calling Claude. The correlation-service uses Mongoose upserts keyed on filing_id. _sent_notifications has a unique compound index preventing duplicate emails. Every service can restart, crash, or replay messages without producing duplicate data.
